Aurora

Aurora is an agentic Bazzite / Linux assistant — a desktop AI app (Electron + React + TypeScript) that answers system questions by actually inspecting your machine instead of guessing. A local LLM running through Ollama decides which read-only inspection tools to call; the app runs them safely and feeds the results back, grounding every answer in your real system state.

How it works

  • Tool-calling loop — an orchestrator drives the model through a curated set of read-only tools (gpu_info, disk_usage, os_status via rpm-ostree), feeding tool output back until it can answer.
  • Bazzite-aware persona — a system prompt grounds the model in atomic-Fedora / Bazzite realities (Flatpak, rpm-ostree layering, Distrobox, Homebrew — never editing the read-only base).
  • Provider-agnostic model adapter — a clean LLM interface with a local Ollama implementation, unit-tested without a running model.
  • Persistence — conversations, messages, a machine profile, and a tool audit trail stored in SQLite.

Architecture

  • Shell: Electron with a clean main / preload / renderer split; the renderer is a React + TypeScript chat UI.
  • Main process: owns the tool registry, the agent orchestrator, the Ollama adapter, command execution, and the SQLite database.
  • Safe command runner: tools shell out through a hardened executor — no shell interpolation, with timeouts and output truncation.
  • Bridge: a small typed window.aurora API exposed over contextBridge — the renderer never touches Node directly.

Security model

Aurora follows the correct Electron posture — contextIsolation: true, nodeIntegration: false, sandbox: true — and goes further on the agent side: the model can only pick among curated, fixed read-only commands, never an arbitrary shell, and a permission gate auto-runs only the safe ones while flagging risky actions for confirmation. Node, the filesystem, command execution, and the database all stay in the main process.
